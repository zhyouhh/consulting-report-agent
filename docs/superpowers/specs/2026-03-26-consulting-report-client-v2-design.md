# 咨询报告客户端 V2 设计稿

## 1. 背景

当前项目的定位是把“咨询报告 Skill”包装成一个可分发给同事使用的桌面客户端，让不会用 Skill 的同事也能完成项目创建、协作写作、阶段推进、质量检查和草稿导出。

现状存在四个核心问题：

1. 客户端内置的是较早期的咨询报告 prompt 资产，已经落后于当前自用的 `consulting-report-skill` 仓库。
2. 产品文案承诺了导出 Word/PDF，但代码层并没有真正实现完整的导出链路。
3. 默认使用方式仍然要求用户手动配置 API，不符合“发给同事开箱即用”的目标。
4. 当前界面只是“项目列表 + 聊天 + 文件预览”的薄壳，阶段状态和质量门禁主要停留在 prompt 里，没有真正产品化。

本次设计的目标不是重做一个在线平台，而是在保留桌面分发形态的前提下，把它收敛成一个稳定、清晰、可共享的 Windows 桌面工作台。

## 2. 目标

### 2.1 业务目标

1. 让同事拿到压缩包后，解压并双击即可使用，不需要自行构建前后端。
2. 默认提供可直接使用的模型通道，同时保留自定义 API 的高级入口。
3. 将新版 `consulting-report-skill` 作为客户端内置运行资产，避免维护两套分裂的写作规则。
4. 把“项目计划、阶段推进、质量检查、可审草稿导出”做成真实产品能力，而不是只写在 prompt 里。

### 2.2 成功标准

1. Windows 用户可通过 `咨询报告助手.exe` 完成从新建项目到导出可审草稿的完整流程。
2. 普通用户无需填写 API Key 即可通过默认通道使用。
3. 高级用户可以切换到自定义 API，并使用已有的模型拉取与模型切换能力。
4. 客户端内项目模板、系统提示、质检脚本、导出脚本与新版 skill 保持一致。

## 3. 非目标

第一阶段明确不做以下事项：

1. 不做 macOS 正式支持，macOS 兼容作为后续版本处理。
2. 不做账号体系、多人协作、云同步和权限系统。
3. 不做最终 Word 终排版，也不承诺一次性输出正式交付版文档。
4. 不做 PPT 交付链、演示稿系统和复杂报表后台。
5. 不把当前项目重构成纯在线 Web SaaS。

## 4. 产品形态

### 4.1 分发形式

第一阶段产品仍为 Windows 桌面客户端，基于现有技术栈延续：

- 后端：FastAPI
- 前端：React
- 桌面壳：PyWebView
- 打包：PyInstaller

分发方式为：

1. 在开发机上构建前端并打包。
2. 产出完整目录 `dist/咨询报告助手/`。
3. 将整个目录压缩为 zip 分发。
4. 同事解压后双击 `咨询报告助手.exe` 启动。

用户不需要安装 Python、Node.js 或手动运行构建命令。

### 4.2 用户定位

- 默认用户：不会配置模型 API、只希望“打开就能写报告”的内部同事。
- 高级用户：知道如何使用 OpenAI 兼容接口，希望自定义模型和供应商。

### 4.3 产品定位

客户端不是通用聊天工具，而是“咨询报告项目工作台”。主产物是：

- `plan/` 项目过程资产
- 报告正文 Markdown
- 审查记录
- 可审草稿导出物

文案上只承诺 `docx` / `pdf` 的“可审草稿/预览导出”，不承诺最终终排版成品。

## 5. 默认通道与自定义 API 设计

### 5.1 双模式

客户端设置界面并列展示两种模式，支持随时切换：

1. 默认通道（推荐，开箱即用）
2. 自定义 API（高级配置）

### 5.2 默认通道

默认通道使用你自有服务器上的“薄中转 OpenAI 兼容接口”，链路为：

`桌面客户端 -> 默认代理接口 -> newapi -> 实际模型`

该默认代理接口需要满足：

1. OpenAI 兼容，至少支持 `chat/completions`。
2. 可选支持 `models`，便于客户端展示默认模型信息。
3. 在代理层强制限制默认模型为 `gemini-3-flash`。
4. 支持基础限速、额度控制、日志与快速停用。
5. 客户端不内置真实上游 key，只内置默认代理地址。

这样做的收益是：

1. 普通用户无需配置即可使用。
2. 真实上游凭证不放进客户端包中。
3. 将来更换上游 key、限流或停用默认通道时，无需重发客户端。

### 5.3 自定义 API

高级用户可切换为自定义 API 模式，并自行配置：

- `api_base`
- `api_key`
- `model`

继续沿用现有模型列表获取逻辑：当用户填写 `api_base + api_key` 后，可调用 `models.list` 获取模型下拉列表。

### 5.4 本地配置模型

客户端本地设置应包含以下字段：

- `mode`: `managed` | `custom`
- `managed_base_url`
- `managed_model`
- `custom_api_base`
- `custom_api_key`
- `custom_model`

界面文案要求：

- 默认通道：`推荐，开箱即用`
- 自定义 API：`高级配置，自行承担可用性`

## 6. Skill 集成设计

### 6.1 总体原则

客户端负责交互、项目管理和可视化；新版 `consulting-report-skill` 仓库负责写作规则、模块路由、模板、质量检查和可审草稿导出。

不能继续维护“客户端一套旧 skill + 个人仓库一套新 skill”的双轨结构。

### 6.2 客户端内置资产

客户端运行时应内置新版 `consulting-report-skill` 的以下资产：

- `SKILL.md`
- `modules/`
- `plan-template/`
- `scripts/quality_check.ps1`
- `scripts/export_draft.ps1`
- 必要时补充 `lib/` 中被脚本依赖的最小集合

第一阶段不必将以下内容打入客户端分发包：

- `tests/`
- `evals/`
- 大部分开发文档

这些内容保留在源 skill 仓库中即可。

### 6.3 上下文恢复

当前客户端只恢复少量项目上下文，不足以支撑新版 skill 的阶段门禁。V2 需要在构建系统提示和恢复上下文时纳入新版核心项目文件，例如：

- `project-overview.md`
- `progress.md`
- `stage-gates.md`
- `notes.md`

目标是让“阶段推进、任务回写、质量门禁”变成真实工作流，而不是 prompt 中的空约束。

### 6.4 项目模板

新建项目时不再使用当前旧模板，改为使用新版 skill 的 `plan-template/` 生成项目结构与基础文件。

## 7. 界面与工作流设计

### 7.1 总体布局

保留当前三栏骨架，但把右侧从纯文件预览升级为“阶段 + 文件”的工作区：

1. 左侧：项目列表
2. 中间：协作聊天
3. 右侧：阶段视图 / 文件视图切换

### 7.2 项目创建

新建项目时至少收集以下信息，并写入新版 `plan/` 模板：

- 项目名称
- 项目类型
- 目标读者
- 截止日期
- 篇幅预期
- 已有材料或备注

### 7.3 对话协作

聊天区保留流式输出能力，但其职责从“泛用聊天”改为“围绕当前项目阶段推进协作”。聊天顶部应显示：

- 当前项目名称
- 当前阶段
- 当前通道（默认通道 / 自定义 API）

### 7.4 阶段视图

阶段视图需要展示：

- 当前阶段
- 已完成项
- 待完成项
- 下一步建议动作

阶段状态应与项目文件中的阶段信息联动，而不是由前端硬编码。

### 7.5 文件视图

保留 Markdown 预览能力，用于查看项目文件、计划文件和正文内容。

### 7.6 质量检查

新增“运行质量检查”操作：

1. 后端调用新版 skill 内置的质量检查脚本，或等价实现。
2. 前端展示分级结果：高风险 / 中风险 / 低风险。
3. 检查结果要可读，不允许只停留在命令行输出。

### 7.7 导出可审草稿

新增“导出可审草稿”操作：

1. 后端调用新版导出脚本生成 `docx`。
2. 前端明确文案为“导出可审草稿”。
3. 导出成功后向用户展示输出目录。
4. 不使用“导出正式 Word 报告”之类夸大表述。

## 8. 平台策略

第一阶段只正式支持 Windows。

原因：

1. 当前项目已有 Windows 打包链路。
2. 同事使用场景以 Windows 为主。
3. macOS 虽然架构上可兼容，但需要单独的 `.app` 打包和后续签名/公证策略。

后续版本可考虑：

- `V1.1`：补充 macOS `.app` 打包和基础测试
- 后续再评估签名与公证

## 9. 实施范围

### 9.1 第一阶段必须完成

1. 双模式配置：默认通道 + 自定义 API
2. 默认代理接入
3. 新版 skill 资产内置
4. 新版模板切换
5. 关键上下文恢复改造
6. 阶段视图和文件视图
7. 质量检查按钮
8. 可审草稿导出按钮
9. Windows 打包与使用说明更新

### 9.2 第一阶段明确不做

1. macOS 正式支持
2. 云同步与多人协作
3. 账号系统
4. 最终终排版
5. PPT 交付流程
6. 服务端复杂后台

## 10. 推荐实施顺序

建议按以下顺序推进，以减少返工：

1. 统一配置模型与设置结构，明确 `managed/custom` 双模式
2. 接入默认薄中转 OpenAI 兼容接口
3. 切换到新版 `consulting-report-skill` 资产与模板
4. 补齐上下文恢复逻辑
5. 实现阶段视图、质量检查、可审草稿导出
6. 更新打包链路和分发文档

这样可以先稳定“通道”和“运行资产”，再做界面和导出能力，避免在错误基础上反复补丁。

## 11. 风险与约束

1. 默认代理服务需要你自己维护可用性；它挂了，普通用户的默认通道就不可用。
2. 可审草稿导出依赖外部工具链时，需要在打包和运行说明中写清依赖边界。
3. 旧项目与新模板之间可能存在兼容问题，需要在实施阶段决定迁移策略。
4. 第一阶段不应夸大导出能力，否则会再次出现“文案已承诺，系统没做到”的问题。

## 12. 决策摘要

本设计选择的是一条“保留桌面交付形态，但把默认通道和运行资产做扎实”的路线：

1. 继续发 Windows exe，而不是转向重型在线平台。
2. 默认通道通过你自己的 OpenAI 兼容薄中转提供开箱即用体验。
3. 高级用户仍可切换为自定义 API。
4. 客户端内置新版 `consulting-report-skill`，不再维护旧 skill 副本。
5. 第一阶段把重点放在项目工作流、质量检查和可审草稿导出，而不是追求全能。
