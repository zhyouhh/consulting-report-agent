# 聊天输入区与待发送附件改造设计稿

## 1. 背景

当前聊天输入区有三个直接影响使用体验的问题：

1. 底部输入框仍然是单行 `<input>`，长消息无法自然换行，也无法随着内容增长而扩高。
2. 附件相关交互割裂：`+` 和拖拽会立刻导入项目材料库，而用户希望 `Ctrl+V`、`+`、拖拽都先进入“待发送附件区”，等点击发送时再统一处理。
3. 附件在发送前几乎没有可视预览，尤其是图片截图无法快速确认内容，文档也无法一眼分辨“这轮临时附件”和“会进入项目材料库的文档”。

这会导致两个现实后果：

1. 用户输入长消息时阅读和修改成本很高。
2. 用户在调 bug、贴截图、补充文档时，很难预测“这个附件发出去后会不会进入项目材料库”。

## 2. 目标

本次改造只解决一个子系统：把当前底部输入区升级为一个多行聊天 composer，并引入“待发送附件队列”。

成功标准：

1. 输入框改成多行 `textarea`，支持 `Enter` 发送、`Shift+Enter` 换行。
2. 输入框高度会随内容自动增长，最多约 `6` 行，再改为内部滚动。
3. `Ctrl+V`、`+` 选文件、拖拽文件全部统一进入“待发送附件区”，点击发送时才真正处理。
4. 图片在待发送区显示真实缩略图卡片，并默认标记为 `本轮临时`。
5. 文档在待发送区显示紧凑文档卡片，并默认标记为 `发送前入库`。
6. 文档在发送前导入项目材料库并附带到本轮消息；若 chat 失败，已入库结果保留并直接参与重试；图片默认只服务本轮，不进入项目材料库。
7. 项目材料选择器继续存在，但与“待发送附件区”明确分层，不混成一套状态。

## 3. 非目标

本次不做：

1. 不重做整个聊天区布局。
2. 不引入新的“临时附件存储目录”或独立后端文件系统。
3. 不把所有附件都改成统一长期入库。
4. 不在这次改造里顺手重构 `ChatPanel` 全部状态管理。
5. 不新增附件批注、重命名、排序、OCR 等增强能力。

## 4. 已确认的交互约定

这部分是本次设计的固定输入，不再反复讨论：

1. 输入行为：
   - `Enter` 发送
   - `Shift+Enter` 换行
2. 输入框版式：
   - 采用 `A` 方案，即“待发送附件区”放在输入框上方
3. 附件进入方式：
   - `Ctrl+V`
   - `+` 选文件
   - 拖拽文件
   - 三者全部统一进入“待发送附件区”
4. 待发送附件的默认归属：
   - 图片：`本轮临时`
   - 文档：`发送前入库`
5. 预览样式：
   - 图片：中等卡片，显示真实缩略图
   - 文档：紧凑卡片，显示文件图标和文件名
6. 输入框扩高上限：
   - 最多约 `6` 行

## 5. 核心方案

### 5.1 Composer 总体布局

底部 composer 调整为三层结构：

1. 待发送附件区
2. 多行输入框
3. 操作区（`+`、发送/停止、简短提示）

布局原则：

1. “待发送附件区”与“项目材料选择区”必须分层。
2. 用户应当一眼区分：
   - 这是项目里已经存在的材料
   - 这是本轮准备发送的附件
3. 附件状态标签必须直接可见，不隐藏在 hover 中。

### 5.2 待发送附件队列

前端新增一个本地状态队列 `pendingAttachments`，只服务当前 composer，不进入全局项目状态。

每个待发送附件至少包含：

- `id`
- `file`
- `displayName`
- `mimeType`
- `kind`：`image` / `document`
- `deliveryMode`：`ephemeral` / `persist`
- `previewUrl`：仅图片需要

默认规则：

1. 图片进入队列时，`kind = image`，`deliveryMode = ephemeral`
2. 文档进入队列时，`kind = document`，`deliveryMode = persist`

### 5.3 图片与文档的落地差异

#### 图片

图片默认只服务当前这轮消息。

实现口径：

1. 发送前不进入项目材料库。
2. 发送时直接作为本轮临时附件传入 chat 请求。
3. 发送成功后不写入 `materials.json`，不进入材料面板。
4. 历史会话重新加载时，不尝试恢复这些临时图片。

这符合用户对截图的心智模型：它更像临时沟通证据，而不是长期项目材料。

#### 文档

文档默认在发送前转成正式项目材料。

实现口径：

1. 发送前先调用现有材料上传接口。
2. 上传成功后，拿到新的 `material.id`。
3. 将这些 `material.id` 自动并入本轮 `attached_material_ids`。
4. 上传一旦成功，文档就已经进入项目材料库，并能在后续轮次继续复用。

这是一条单一真相：文档入库发生在 chat 请求之前，而不是 chat 成功之后。

这符合文档型材料的使用方式：往往不止服务一轮，而是整个报告过程都可能反复引用。

## 6. 前后端数据流

### 6.1 前端发送前的分流

点击发送后，composer 先做附件分流：

1. `pendingAttachments` 中的文档：
   - 先调用 `/api/projects/{project_id}/materials/upload`
   - 成功后得到 `material.id`
   - 并入本轮 `attached_material_ids`
2. `pendingAttachments` 中的图片：
   - 在前端转成 `data_url`
   - 打包进本轮 `transient_attachments`

因此，本轮 chat 请求包含两类附件：

1. 长期材料：`attached_material_ids`
2. 临时图片：`transient_attachments`

文档型附件的事务边界固定如下：

1. 先上传入库
2. 再发 chat
3. 如果 chat 失败，已上传文档仍然保留在项目材料库中
4. 重试时不得重复上传同一批已成功入库的文档

### 6.2 后端请求扩展

现有 `ChatRequest` 只支持 `attached_material_ids`，不足以表达“本轮临时图片”。

本次只做一个薄扩展：

- 新增 `transient_attachments`

每个临时附件至少包含：

- `name`
- `mime_type`
- `data_url`

当前仅允许图片类型进入 `transient_attachments`。

### 6.3 ChatHandler 的拼装逻辑

`ChatHandler` 在构造 provider user message 时，需要同时合并：

1. 项目材料 `attached_material_ids`
2. 临时图片 `transient_attachments`

拼装规则：

1. 文本材料仍通过现有材料机制暴露给模型，需要时调用 `read_material_file`
2. 项目中的图片材料仍走现有 `data_url` 注入逻辑
3. 本轮临时图片直接作为额外的 `image_url` 内容块注入本轮 user message

### 6.4 会话持久化边界

本轮临时图片不进入长期会话重建链路。

这意味着：

1. 同一轮内，模型能看到图片
2. 本轮结束后，历史对话只保留文字消息
3. 后续轮次和重开项目时，不再自动把这些图片重新附带给模型

这是有意为之，不是遗漏。

## 7. UI 细节

### 7.1 图片卡片

图片卡片是中等尺寸，至少满足：

1. 能看出大致截图内容
2. 能清楚点击删除
3. 能直接看到状态标签 `本轮临时`

它不需要做到图片编辑器级别的预览，只要足够让用户确认“贴对了图”。

### 7.2 文档卡片

文档卡片是紧凑样式，至少包含：

1. 文件图标
2. 文件名
3. 扩展名或类型
4. 状态标签 `发送前入库`
5. 删除按钮

它的目标是节省空间，而不是提供内容预览。

### 7.3 项目材料选择区

现有项目材料选择器保留，但语义要更明确：

1. 它表示“项目里已有的材料，可选择附带到本轮”
2. 它不等同于“这轮刚刚新增的待发送附件”

必要时可增加极短说明文案，但不要把输入区做成表单。

### 7.4 输入行为细则

输入框行为固定如下：

1. `Enter` 发送
2. `Shift+Enter` 换行
3. IME composing（中文输入法候选确认）状态下，`Enter` 只确认输入，不触发发送

## 8. 错误处理与边界

### 8.1 无项目时

无 `projectId` 时：

1. 禁用 `+`
2. 禁用拖拽
3. 粘贴文本仍允许进入输入框
4. 粘贴中的附件部分忽略，并提示“请先选择或创建项目后再附加附件”

### 8.2 模型不支持图片时

如果待发送队列里有图片，而当前模型按保守规则不支持图片：

1. 发送前直接拦截
2. 提示用户切换模型或删除图片
3. 保留待发送队列，不自动清空

这里的“保守规则”不是新造规则，而是直接复用前端现有的 `supportsImageAttachments` 能力判断：

1. 默认通道一律视为支持图片
2. 自定义模型按当前 `modelCapabilities` 里的保守白名单判断

### 8.3 上传/发送失败时

如果文档上传失败或临时图片处理失败：

1. 本轮消息不发送
2. 待发送附件队列保留
3. 用户可删除、调整后重试

如果 chat 请求失败：

1. 用户消息和助手占位按现有错误逻辑处理
2. 临时图片仍保留在待发送附件队列里
3. 已经上传成功的文档不再留在待发送附件队列里，而是转成已选项目材料，供用户直接重试
4. 重试时不得重复上传这批已成功入库的文档

### 8.4 成功后的清理规则

发送成功后：

1. 清空输入框
2. 清空待发送附件队列
3. 文档型附件如果已在发送前入库，自动合并到现有材料列表并维持本轮附带关系
4. composer 发送成功后恢复干净状态，新入库文档进入材料列表，但不在下一轮继续保持预选中

### 8.5 粘贴混合内容

如果一次粘贴同时包含文本和图片：

1. 文本进入输入框
2. 有项目时，图片进入待发送附件队列
3. 无项目时，文本仍进入输入框，图片忽略并提示

有项目时不丢文本也不丢图片；无项目时不丢文本，但图片会被忽略并提示，不要求用户二次操作。

## 9. 测试要求

### 9.1 前端

至少覆盖：

1. `pendingAttachments` 队列的新增、删除、清空
2. 图片默认 `ephemeral`
3. 文档默认 `persist`
4. `Enter` 发送、`Shift+Enter` 换行
5. IME composing 状态下，`Enter` 只确认输入，不触发发送
6. 发送成功后清空待发送队列
7. chat 失败时，临时图片保留，已入库文档转成已选项目材料而不是重复留在队列里
8. 文档先上传、再合并 `attached_material_ids`
9. 图片进入 `transient_attachments`

### 9.2 后端

至少覆盖：

1. `ChatRequest` 新增 `transient_attachments` 的兼容解析
2. `ChatHandler` 能把长期材料与临时图片同时拼进 provider message
3. 临时图片不进入项目材料库
4. 临时图片不污染长期对话重建逻辑

## 10. 方案取舍

本次不采用“新建临时附件存储目录”的方案，原因很简单：

1. 会把一次性截图长期落盘，污染项目目录
2. 会为一个临时沟通场景引入新的存储生命周期
3. 复杂度和收益不匹配

也不采用“所有附件都统一先入库”的方案，因为这会把大量截图垃圾堆进项目材料库，违背用户使用习惯。

因此，本次采用最朴素也最稳的双通道：

1. 图片：本轮临时
2. 文档：发送前入库

这就是当前需求下最小、最清楚、最不容易出错的实现。
