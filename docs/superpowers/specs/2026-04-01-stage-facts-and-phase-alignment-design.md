# 项目阶段、事实源与模板对齐设计稿

## 1. 背景

当前客户端在“项目信息文件”和“阶段状态”上同时存在多套事实源，已经开始直接影响使用体验和调试效率：

1. 项目目录里同时存在 `project-info.md` 和 `project-overview.md`，但 v2 实际已经切到 `project-overview.md` 作为主信息文件。
2. 文件预览默认仍打开 `project-info.md`，导致用户看到一个空模板，误以为项目基本信息没有写入。
3. 项目目录里可能出现模型自行创建的 `plan/gate-control.md`，但系统并不会读取它，形成“看起来像阶段控制文件、实际上完全无效”的游离文件。
4. 阶段模板 `stage-gates.md` 定义的是完整 `S0 -> S7` 顺序阶段，但后端实际只做了一个粗糙的 `S0 / S1 / S4` 三段式推断。
5. `stage-gates.md` 中列出的部分关键文件，如 `data-log.md`、`analysis-notes.md`、`review-checklist.md`，当前模板目录里并不存在，导致“模板定义”和“模板集合”本身也不一致。

这会造成三个直接后果：

1. 用户看到的“项目基本信息”与系统真正使用的上下文文件不一致。
2. 顶部阶段、右侧阶段面板、项目目录中的阶段相关文件可能彼此打架。
3. 模型在约束不清时会自行发明 `plan/*.md` 文件，进一步污染项目目录并放大阶段混乱。

## 2. 目标

本次改造只解决一个子系统：统一项目信息文件、正式 `plan/` 文件集合、阶段推断逻辑和 UI 展示口径，使它们全部围绕同一套规则运行。

成功标准：

1. `project-overview.md` 成为唯一主信息文件。
2. `project-info.md` 退出正式工作流，不再作为默认预览和阶段依据。
3. `plan/` 的正式文件集合明确、固定，模型不能再写入未登记的游离文件。
4. 在允许写 `outline.md` 和 `research-plan.md` 之前，必须先完成一轮轻量边界摸底，并把依据落盘到正式文件。
5. 阶段推断补齐到完整 `S0 -> S7`，且顺序语义与产品设计一致。
6. `S6 演示准备` 变成可选阶段，仅在项目明确需要演示时启用。
7. 后端、阶段模板、UI 显示统一依赖一套事实源，不再出现 `gate-control.md` 这类“看起来像规则、其实完全不生效”的旁路文件。

## 3. 非目标

本次不做：

1. 不重写整个 Skill 提示词体系。
2. 不把所有阶段都改成完全由模型显式声明推进。
3. 不自动解析并迁移旧项目中自由文本 `project-info.md` 的全部内容。
4. 不在这次改造里顺手重做报告正文工作流或导出工作流。

## 4. 已确认决策

这部分作为本次实现的固定输入：

1. 阶段模型必须完整保留 `S0 -> S7`。
2. 不能再出现 `S1` 直接跳 `S4` 的默认逻辑。
3. 在写 `outline.md` 和 `research-plan.md` 之前，模型必须先做一次轻量边界摸底。
4. 这轮摸底可以来自项目材料，也可以来自 `web_search + fetch_url`，但依据必须先落盘到正式文件。
5. `S1 -> S2` 的进入条件不是“有大纲就行”，而是 `outline.md` 和 `research-plan.md` 都真实落盘。
6. `S6 演示准备` 不是所有项目都必须经过，应作为可选阶段。
7. `project-info.md` 不再保留为正式主信息文件，但可以参考其字段补强 `project-overview.md` 模板。

## 5. 单一事实源设计

### 5.1 主信息文件

正式主信息文件统一为：

- `plan/project-overview.md`

它承担以下职责：

1. 项目基础信息
2. 核心问题
3. 项目范围
4. 成功标准
5. 与交付形态有关的开关信息

`project-info.md` 视为遗留文件：

1. 不再作为默认预览文件
2. 不再进入 `CORE_CONTEXT_FILES`
3. 不再参与阶段推断
4. 新项目不再主动初始化或填充它

### 5.2 阶段唯一事实源

阶段对外统一呈现为 `stage-gates.md` 中的内容，但其“当前阶段”和自动勾选项不再由模型随意写入，而是由后端根据正式文件状态回填。

也就是说：

1. UI 顶部当前阶段
2. 右侧阶段面板
3. `stage-gates.md` 中的当前阶段与自动勾选项

都应来自同一套后端推断结果。

### 5.3 游离文件处理

`plan/gate-control.md` 这类未登记文件不是正式事实源。

处理原则：

1. 系统不读取它
2. 新版本禁止模型继续创建它
3. 旧项目里如果已经存在，允许保留为历史残留，但不参与任何阶段与上下文计算

## 6. 正式模板文件集合

### 6.1 保留的正式模板

以下文件继续作为正式 `plan/` 文件：

- `project-overview.md`
- `progress.md`
- `stage-gates.md`
- `notes.md`
- `outline.md`
- `research-plan.md`
- `references.md`
- `tasks.md`
- `review.md`

### 6.2 新增并补齐的正式模板

为让八阶段真正闭环，需要补齐下列正式模板文件：

- `data-log.md`
- `analysis-notes.md`
- `review-checklist.md`
- `presentation-plan.md`
- `delivery-log.md`

### 6.3 移除出正式工作流的文件

以下文件不再属于正式工作流：

- `project-info.md`

它不一定要立刻从所有旧项目物理删除，但应从：

1. 新项目默认初始化入口
2. 默认预览入口
3. 阶段判定
4. 核心上下文

中全部退出。

## 7. `project-overview.md` 模板优化

`project-overview.md` 需要吸收原本只有 `project-info.md` 才表达的有用字段，避免“废掉 info 后信息容量反而下降”。

建议补强后的 `project-overview.md` 至少包含：

1. `项目名称`
2. `报告类型`
3. `报告主题`
4. `项目背景`
5. `目标读者`
6. `预期篇幅`
7. `交付时间`
8. `特殊要求`
9. `交付形式`
10. `成功标准`

其中 `交付形式` 用于控制 `S6` 是否启用，推荐值：

- `仅报告`
- `报告+演示`

默认值设为：

- `仅报告`

这样“不复杂、不需要演示”的项目可以自然跳过 `S6`。

## 8. 八阶段语义与进入条件

### 8.1 总体原则

后端不再只看“当前有无 outline / report”，而是按顺序推断“已完成到哪个阶段、当前处于哪个阶段”。

推断口径：

1. 只使用正式文件集合做自动推断
2. 自动推断只依赖“有意义内容”，不以模板空文件为完成
3. 当前阶段定义为“首个尚未完成但应该执行的阶段”
4. 对可选阶段 `S6`，若项目不要求演示，则视为跳过而不是阻塞

### 8.1.1 大纲前置摸底门禁

在允许写 `outline.md` 和 `research-plan.md` 之前，必须先完成一次轻量边界摸底。

摸底来源允许两种：

1. 项目已有材料
2. `web_search` + `fetch_url`

模型可以自己判断是否需要联网，但后端不接受“没有依据、直接编大纲”的路径。

最小证据门槛定义为：

1. 至少 `2` 份可读项目材料
2. 或至少 `2` 个外部有效来源
3. 或 `1` 份项目材料 + `1` 个外部有效来源

而且这些依据必须先写入正式文件：

1. `notes.md`：记录主题边界、明确不做范围、当前假设、关键判断
2. `references.md`：记录来源列表与简要摘要

未满足上述门禁时，本轮只允许：

1. 继续提问
2. 搜集资料
3. 写 `notes.md`
4. 写 `references.md`

未满足门禁时，禁止写：

1. `outline.md`
2. `research-plan.md`

### 8.2 各阶段定义

#### S0 项目启动

目标：项目立项、边界和基本交付信息完成初始化。

自动完成证据：

- `project-overview.md` 已写入有效项目内容

#### S1 研究设计

目标：完成边界摸底、形成研究结构与研究计划。

自动完成证据：

- `notes.md` 有效
- `references.md` 有效
- `outline.md` 有效
- `research-plan.md` 有效

说明：

1. 只有摸底笔记，没有大纲/研究计划，仍然属于 `S1 进行中`
2. 只有大纲，没有研究计划，仍然属于 `S1 进行中`
3. 只有研究计划，没有大纲，也仍然属于 `S1 进行中`
4. 两者都完成后，进入 `S2`

#### S2 数据收集

目标：完成资料收集与事实记录。

自动完成证据：

- `data-log.md` 有效

#### S3 分析洞察

目标：把原始资料转为结构化洞察。

自动完成证据：

- `analysis-notes.md` 有效

#### S4 报告撰写

目标：形成正文草稿。

自动完成证据：

- `report_draft_v1.md`
- 或 `content/report.md`
- 或 `content/draft.md`
- 或 `output/final-report.md`

中至少一个存在且有有效内容

说明：

正文产物继续允许放在 `content/` 或根目录草稿中，不强制塞回 `plan/`。

#### S5 质量审查

目标：完成对正文的审查与修订闭环。

自动完成证据：

- `review-checklist.md` 有效

可选辅助文件：

- `review.md`

其中 `review.md` 用于记录审查意见，不单独作为阶段完成依据。

#### S6 演示准备（可选）

目标：为汇报场景准备 PPT / 演讲提纲 / Q&A。

仅当 `project-overview.md` 的 `交付形式 = 报告+演示` 时启用。

自动完成证据：

- `presentation-plan.md` 有效

若项目交付形式为 `仅报告`，则 `S6` 自动视为跳过，不阻塞进入 `S7`。

#### S7 交付跟进

目标：记录最终交付与后续跟进。

自动完成证据：

- `delivery-log.md` 有效

## 9. “有意义内容”判定

所有阶段推断都必须区分“模板空文件”和“真实内容”。

统一规则：

1. 文件不存在：无效
2. 文件内容为空：无效
3. 文件内容与模板归一化后完全一致：无效
4. 只有标题、没有实体内容：无效
5. 存在用户/模型填充后的真实内容：有效

对部分文件可增加更具体的最低标准：

1. `notes.md` 至少包含主题边界、排除范围、假设或关键判断中的两类
2. `references.md` 至少包含满足最小证据门槛的来源记录
3. `outline.md` 至少包含多个章节标题
4. `research-plan.md` 至少包含研究方法/数据来源/执行步骤之一
5. `data-log.md` 至少包含一条具体数据来源、访谈记录或资料条目
6. `analysis-notes.md` 至少包含一条洞察或分析结论
7. `review-checklist.md` 至少包含一次具体检查记录
8. `presentation-plan.md` 至少包含演示结构或讲述安排
9. `delivery-log.md` 至少包含一次交付或反馈记录

## 10. 阶段面板与模板文案调整

`stage-gates.md` 需要与实际阶段推断规则对齐。

原则：

1. 保留 `S0 -> S7` 展示结构
2. 每个阶段至少要有一个系统可自动识别的正式文件证据
3. 手工项可以保留，但不应成为自动阶段推进的唯一依据

推荐调整：

1. `S1` 明确写入 `outline.md 完成`、`research-plan.md 完成`
2. `S2` 明确写入 `data-log.md 更新`
3. `S3` 明确写入 `analysis-notes.md 创建/更新`
4. `S5` 明确写入 `review-checklist.md 完成`
5. `S6` 明确写入 `presentation-plan.md 完成`
6. `S7` 明确写入 `delivery-log.md 更新`

## 11. UI 调整

### 11.1 文件预览默认文件

默认预览文件从：

- `plan/project-info.md`

改为：

- `plan/project-overview.md`

### 11.2 文件列表呈现

`project-info.md` 不再作为默认入口。

处理策略：

1. 新项目不出现它
2. 旧项目如遗留存在，可继续显示在文件列表中，但不作为默认打开项

### 11.3 阶段显示

顶部当前阶段与右侧阶段面板都只使用 `/api/projects/{id}/workspace` 返回的 `stage_code/status/completed_items/next_actions`。

前端不自行根据文件名或聊天内容猜阶段。

## 12. 写文件约束

后端 `write_file` 对 `plan/` 增加正式白名单。

允许写入的 `plan` 文件仅限正式模板集合与其兼容集合。

对白名单外文件：

1. 拒绝写入
2. 返回清晰错误信息
3. 引导模型改写到正式文件中

这将直接阻止：

- `gate-control.md`
- 其他临时发明的 `plan/*.md`

同时，对白名单内的关键文件增加阶段门禁：

1. 未满足摸底门禁时，拒绝写 `outline.md`
2. 未满足摸底门禁时，拒绝写 `research-plan.md`
3. 错误信息要明确指出：请先基于材料或检索结果更新 `notes.md` 与 `references.md`

## 13. 旧项目兼容与迁移

### 13.1 旧项目保留策略

旧项目中已经存在的：

- `project-info.md`
- `gate-control.md`

不做强制删除。

但新版本中：

1. 不再把它们当正式事实源
2. 不再默认预览它们
3. 不允许继续新建类似游离文件

### 13.2 旧项目阶段修复

旧项目进入工作区时，应重新根据正式文件状态回算阶段，并覆盖 `stage-gates.md` 中的当前阶段、自动勾选项和更新日期。

这样即使目录里残留 `gate-control.md`，系统也能回到统一阶段口径。

## 14. 测试要求

至少补齐以下测试：

### 14.1 后端

1. 新项目初始化后默认阶段为 `S0`
2. 未满足摸底门禁时，写入 `outline.md` 被拒绝
3. 未满足摸底门禁时，写入 `research-plan.md` 被拒绝
4. 已满足摸底门禁但只有 `outline.md` 时，阶段仍为 `S1`
5. `outline.md + research-plan.md` 同时有效时，阶段进入 `S2`
6. `data-log.md` 有效时，阶段进入 `S3`
7. 正文草稿存在时，阶段进入 `S4`
8. `review-checklist.md` 有效时，阶段进入 `S5`
9. `S6` 只在 `交付形式 = 报告+演示` 时启用
10. `presentation-plan.md` 有效时，且项目要求演示，阶段进入 `S6`
11. `delivery-log.md` 有效时，阶段进入 `S7`
12. `project-info.md` 不再参与主上下文和阶段推断
13. 写入 `plan/gate-control.md` 被拒绝

### 14.2 前端

1. 文件预览默认打开 `project-overview.md`
2. 旧项目中存在 `project-info.md` 时，不会默认跳到它
3. 顶部当前阶段与右侧阶段面板跟随后端 `/workspace` 返回

## 15. 方案摘要

本次方案的核心判断是：

1. 不再继续容忍“多套阶段事实源并存”的状态。
2. `project-overview.md` 成为唯一主信息文件，`project-info.md` 正式退役。
3. 大纲和研究计划不是“凭感觉先写”，而是建立在先摸底、先落盘依据的硬门禁之上。
4. 八阶段不是只体现在模板里，而要在代码里真正形成顺序推断。
5. `S6` 保留，但只在项目明确需要演示时启用。
6. 模型能力不再建立在“写哪都行”的宽松文件权限上，而要通过正式 `plan` 白名单与关键阶段门禁约束回到产品设计边界内。
