# Fetch URL HTTP-First 加固设计稿

## 1. 背景

当前项目里的 `fetch_url` 已经接入聊天流程，也能在一部分公开网页上工作，但实际使用体验仍明显偏弱：

1. 对普通官网、文档站、政府站的成功率和正文质量不稳定。
2. 对中文站点的编码处理较弱，容易出现标题或正文乱码。
3. 对 `403`、挑战页、错误页、跳转页的分类过于粗糙，模型只能得到“抓取失败”，难以做下一步判断。
4. 对百度百科这类国内常见资料源，当前实现会因为请求层、跳转层或风控页识别不足而直接失败。
5. 同一轮对话里模型可能多次抓同一页面，当前实现没有轻量缓存，浪费时间与带宽。

与此同时，我们已经明确不希望这一轮走浏览器兜底，也不希望引入“抓完再让小模型总结”的高成本路径。本轮应在保持 `HTTP-first` 架构的前提下，把现有抓取器尽量打磨成一个更可靠的网页正文读取工具。

## 2. 目标

本次改造的核心目标是：在不引入浏览器自动化和二次模型总结的前提下，升级 `fetch_url` 的请求、编码、正文提取和错误分类能力，使其在真实工作流里更稳定、更可控、更少误导。

成功标准：

1. 保持现有 `fetch_url(url)` 工具接口不变，不要求前端联动修改。
2. 提高官网、文档站、政府站的成功率和正文可读性。
3. 明显改善中文站点的编码处理，减少标题与正文乱码。
4. 对百度百科类页面尽量提升命中率，至少不要因为我们自己的请求/跳转/编码处理太弱而轻易失败。
5. 对 `403`、错误页、挑战页、重定向页、非正文页给出更清晰的失败类型。
6. 为重复 URL 增加轻量缓存，避免同一会话中反复抓取。

## 3. 非目标

本次不做：

1. 不引入浏览器兜底，不使用 Playwright、Chrome、Headless 浏览器等方案。
2. 不承诺所有强反爬站都能成功抓取。
3. 不引入“抓取后再用小模型按 prompt 提炼”的高成本链路。
4. 不改 `web_search` 服务来源，不替换搜索后端。
5. 不修改前端交互形态，只允许在后端返回结构里增加兼容字段。
6. 不做 URL 级证据绑定；本轮只判断“本轮是否发生过可用的 `fetch_url` 成功”，不把某条来源与某次抓取逐条强绑定。

## 4. 已确认决策

1. 工具名继续使用 `fetch_url`，参数仍为 `url`。
2. 本轮走 `HTTP-first` 路线，不上浏览器兜底。
3. 允许参考 `claude-code` 和 `opencode` 的做法，但只吸收适合当前项目的抓取层增强，不照搬其权限体系或二次模型处理。
4. 百度百科不是“必须 100% 成功”的硬承诺，但会作为明确优化目标，而不是顺带碰运气。
5. 聊天层中“`web_search` 后写正式文件前应优先 `fetch_url`”的约束继续保留，并与本轮抓取器增强保持一致。

## 5. 现状问题拆解

### 5.1 请求层过薄

当前 `fetch_url` 使用的是非常轻量的直接请求：

1. 请求头较少，站点更容易判断为脚本抓取。
2. 对重定向、最终 URL、不同站点的返回壳页缺少细化处理。
3. 对 `403` 只报告状态码，没有进一步区分挑战页、风控页、登录限制或普通拒绝。

### 5.2 编码层不稳

当前抓取逻辑对中文页面存在明显缺口：

1. 默认解码路径不足以覆盖常见中文编码场景。
2. 仅依赖响应头时容易失真，标题和正文都可能乱码。
3. 政府站、旧门户站、部分中文资讯页更容易暴露这个问题。

### 5.3 正文提取链路单一

当前正文提取主要依赖单一路径。问题在于：

1. `trafilatura` 对很多现代站点表现不错，但对特殊结构页、脚本较多的中文站、错误页和壳页并不总是稳定。
2. 缺少提取失败后的回退策略，导致“有页面但没正文”和“页面根本不可读”混在一起。

### 5.4 结果语义太弱

模型真正需要的不只是“请求成功/失败”，还需要知道：

1. 这是不是正文页。
2. 这是挑战页还是错误页。
3. 最终 URL 是不是被跳走了。
4. 这次抓到的内容是否值得继续写入正式文件。

当前返回语义不够细，模型容易继续误判。

## 6. 目标方案总览

本次设计把 `fetch_url` 重构为一个六段式流水线，但对外仍只暴露一个工具：

1. URL 预处理
2. 请求层增强
3. 响应判别
4. 编码探测与解码
5. 正文提取与错误页识别
6. 结果标准化与缓存

这六段流水线均落在后端 `backend/chat.py` 内，最大程度控制改动面。

## 7. 详细设计

### 7.1 URL 预处理

新增统一 URL 预处理逻辑，职责包括：

1. 继续校验 `http` / `https` scheme。
2. 保留现有的内网、回环地址、受限主机名拦截。
3. 自动将 `http` 升级为 `https`。
4. 规范化 URL，用于缓存 key 和日志输出。
5. 保留并返回 `original_url` 与 `normalized_url` 的关系，方便出错提示。

### 7.2 请求层增强

请求层升级目标是“更像真实浏览器，但仍然是纯 HTTP 请求”。

增强项：

1. 使用更完整的请求头：
   - `User-Agent`
   - `Accept`
   - `Accept-Language`
   - `Accept-Encoding`
2. 对 `html/text/xhtml+xml/plain` 给出更合理的 `Accept` 优先级。
3. 控制超时、最大响应体大小、最大重定向次数。
4. 记录最终 URL 与中间状态。
5. 为后续缓存保留规范化后的请求维度。

实现上应参考：

1. `opencode` 的浏览器风格 `User-Agent` 与 `Accept` 设计。
2. `claude-code` 的 URL 升级、重定向边界与缓存思路。

### 7.3 重定向处理

本轮不采用“完全自动跟随所有重定向”的粗放策略。

建议规则：

1. 逐跳处理 `301/302/307/308`，不能只在最终结果上做一次校验。
2. 最大跳转次数固定，例如 `5` 或 `8`，避免循环跳转。
3. 默认只自动跟随以下情况：
   - 下一跳 host 与当前 host 完全相同
   - 下一跳 host 仅为当前 host 的 `www.` / 裸域等价变体
   - 在上述 host 白名单内，仅 path / query / fragment 变化
4. 白名单判断不引入注册域名 / 公共后缀表逻辑，不做“同主域大概算安全”的模糊放行。
5. 除 `www.` / 裸域等价变体外，任何跨 host 跳转一律不自动跟随。
6. `http -> https` 升级优先尝试，但若出现“连接失败 / TLS 失败 / 明确协议不可达”这类错误，可回退到原始 `http` 再试一次。
7. 对跨站跳转或不被允许的跳转，返回结构化提示，而不是盲跳。
8. 保留 `final_url` 字段，让模型知道真正读取的是哪个页面。
9. 对重定向循环或超限跳转，返回明确错误。

这可以吸收 `claude-code` 里“安全重定向”思路，但简化为适合当前项目的版本。

### 7.4 编码探测与解码

编码探测是这轮最重要的稳定性改动之一。

建议采用字节级探测顺序：

1. 先检查 BOM
2. 再读响应头里的 `charset`
3. 再从原始 bytes 的前一段内容中扫描 HTML 头部的 `<meta charset>` / `http-equiv`
4. 如仍不确定，再使用 `requests` / `charset-normalizer` 的表观编码
5. 对中文站点增加常见回退编码尝试，例如：
   - `utf-8`
   - `gb18030`
   - `gbk`
   - `gb2312`

要求：

1. 标题和正文共用同一解码结果，避免标题与正文编码不一致。
2. 若响应头与 meta 冲突，优先相信 BOM，其次响应头，其次 meta，再到表观编码。
3. 如果出现“页面成功返回但文字高度乱码”的情况，应明确识别为解码质量问题，而不是直接当正文成功。
4. 乱码判定至少应覆盖：
   - 大量替换字符
   - 标题与正文中异常高密度的不可读字符
   - 中文页面正文基本不含有效汉字但含大量乱码碎片

### 7.5 正文提取

正文提取采用“双层提取”：

1. 第一层保留 `trafilatura`
2. 第二层增加基础回退提取器

回退提取器优先级：

1. 如果仓库已存在适合的 HTML 正文提取库，可优先复用
2. 若没有，则用 `BeautifulSoup` 做轻量主内容提取

回退提取器的目标不是做到“像浏览器一样聪明”，而是：

1. 在 `trafilatura` 没抓出正文时，给出一个更稳的备用结果
2. 避免纯导航壳页、脚本页、错误页被误当成正文

决策顺序必须固定：

1. 先完成解码
2. 再做错误页 / 挑战页 / 壳页识别
3. 若判定为明显错误页，直接返回失败，不进入正文提取
4. 若不是明显错误页，先跑 `trafilatura`
5. 若 `trafilatura` 结果为空或低质量，再跑回退提取器
6. 对最终提取结果做成功门槛判断；不满足门槛时返回“未提取到可用正文”，而不是伪成功

### 7.6 错误页 / 挑战页识别

这轮必须让 `fetch_url` 更会识别“假正文”。

至少需要识别：

1. Cloudflare / challenge / verification 页面
2. 百度系错误页、跳转页、反爬提示页
3. 明显登录墙、访问限制页
4. 内容极短且以跳转脚本、风控提示、错误模板为主的页面

输出策略：

1. 不把这类页面正文直接返回为成功内容
2. 返回更明确的错误消息，例如：
   - `目标网站拦截了自动抓取（403）`
   - `目标网页返回了挑战页，当前 HTTP 抓取无法继续`
   - `抓到的是错误页/壳页，未提取到可用正文`
3. 低置信度时，保守返回 `error_type=non_readable_page`，不要冒险当正文成功

成功判定门槛至少包括：

1. 标题与正文都通过基础解码质量检查
2. 正文长度高于最小阈值，或明确命中高置信内容结构
3. 不包含已知挑战页 / 登录墙 / 错误页信号
4. 不只是纯跳转脚本、空白模板或极短壳内容

### 7.7 结果标准化

保持与现有前端兼容，同时增强结果语义。

成功结果契约必须明确：

1. `status`
2. `title`
3. `content`
4. `url`
   - 保持现有语义与字段名，继续对外返回，避免破坏兼容
   - 语义上保持为“最终实际读取的 URL”，不回退成原始输入 URL
5. `final_url`
   - 新增兼容字段
   - 本轮与 `url` 保持一致，用于后续内部逐步迁移和调试
6. `content_type`
7. `truncated`
8. `error_type` 不出现在成功结果中

失败结果契约必须明确：

1. `status`
2. `message`
3. `url`
   - 若可得，保留原始请求 URL
4. `final_url`
   - 若已发生跳转且可得，返回最终 URL
5. `error_type`
   - 必须存在于可分类错误中，例如：
     - `http_status_403`
     - `http_status_404`
     - `redirect_blocked`
     - `challenge_page`
     - `non_readable_page`
     - `decode_failed`
     - `timeout`
     - `unsupported_content_type`

其中 `error_type` 可用于聊天层未来更细的提示，但本轮不要求前端展示。

### 7.8 缓存

新增轻量缓存，但缓存作用域必须收紧。

原则：

1. key 不能只用 URL，至少还要纳入会影响结果的关键维度，例如：
   - 规范化 URL
   - 项目 / 会话作用域
   - 重要请求模式差异（如是否发生 HTTP 回退）
2. 缓存目标优先解决“同一会话 / 同一轮反复抓”问题，不应让一次瞬时失败污染后续独立任务。
3. 成功结果缓存时间可较长，例如 10-15 分钟。
4. 失败结果不能一概做负缓存。
5. 只有确定性失败才允许短时负缓存，例如：
   - `404`
   - `unsupported_content_type`
   - 明确的 `redirect_blocked`
6. 对易变失败默认不做负缓存，或只做本轮极短去重，例如：
   - `403`
   - `challenge_page`
   - `timeout`
   - `decode_failed`

## 8. 与聊天工作流的关系

本次不改工具名与交互方式，但会让聊天层更容易做对事：

1. 当模型 `web_search` 后调用 `fetch_url`，成功率更高、内容更稳定。
2. 当模型想写 `references.md`、`outline.md`、`research-plan.md` 等正式文件时，`fetch_url` 的结果更适合直接作为依据。
3. 当网页本身不适合 HTTP 抓取时，模型能收到更明确的失败类型，从而换源，而不是反复撞同一站。

## 9. 测试策略

采用 TDD，先补失败测试，再改实现。

至少覆盖以下测试矩阵：

### 9.1 请求与跳转

1. 普通 HTML 页面抓取成功
2. 同 host 跳转允许
3. `www.` 与裸域互转允许
4. 跨 host 跳转被阻止
5. 跳转超限失败
6. `http -> https` 升级成功
7. `https` 失败后回退 `http` 的受控场景

### 9.2 编码与解码

1. 响应头指定编码成功解码
2. `meta charset` 成功解码
3. 中文页面 `gb18030/gbk` 成功解码
4. header / meta 冲突时按既定优先级处理
5. 乱码页被识别为解码失败或不可读页

### 9.3 内容提取与错误页识别

1. `trafilatura` 提取成功
2. `trafilatura` 失败后回退提取成功
3. 挑战页 / 风控页不被当正文
4. 百度百科类错误页不被当正文
5. `text/plain` 直接返回文本
6. 非支持内容类型被正确拒绝
7. 超大响应被截断或拒绝

### 9.4 缓存

1. 成功结果缓存命中
2. 确定性失败短时负缓存
3. `403/challenge/timeout` 不污染后续会话
4. 不同作用域不会错误共享缓存

### 9.5 返回契约

1. 成功结果必须继续保留现有 `url` 字段，且其值保持为最终实际读取的 URL
2. 成功结果必须新增 `final_url` 字段，且本轮与 `url` 保持一致
3. 若发生允许范围内跳转，成功结果中的 `url/final_url` 必须反映最终落点，而不是原始输入
4. 可分类失败必须返回非空 `error_type`
5. 失败结果中的 `url/final_url` 语义与 spec 保持一致

### 9.6 聊天联动

1. `fetch_url` 成功后，同轮允许写正式文件
2. `fetch_url` 失败时不会错误解锁正式写入
3. `web_search` 后未 `fetch_url` 仍会被写文件门禁拦住
4. 本轮只解决“是否发生过成功 `fetch_url`”这一层门禁
5. 本轮不解决 URL 级证据绑定，不强制把某次 `fetch_url` 成功与某条外部来源逐条绑定

建议测试资产：

1. 原始 bytes fixture
2. mock redirect chain fixture
3. 缓存 TTL / negative-cache fixture

## 10. 风险与权衡

### 10.1 仍然无法覆盖所有强反爬站

这是 `HTTP-first` 的天然边界。本轮只能把“本来能抓却因为我们太弱抓不好”的情况尽量修掉，不能承诺彻底解决所有反爬。

### 10.2 中文站点编码修复会增加实现复杂度

但这部分价值很高，因为政府站、旧门户站、百科类页面在中文场景里本来就重要。

### 10.3 回退提取器不能过度复杂

如果回退提取逻辑堆得太多，就会演变成半套浏览器/半套阅读器，反而不好维护。原则上应保持“少量规则 + 清晰失败类型”，而不是无止境打补丁。

### 10.4 实现文件边界需要保持可维护

虽然本轮改动会从 `backend/chat.py` 入口落地，但具体实现允许抽出有限的私有 helper 或同目录抓取辅助模块，以避免 `chat.py` 继续失控膨胀。前提是：

1. 不改变外部接口
2. 不引入新的持久化系统
3. 测试边界比现在更清晰，而不是更散

### 10.5 大响应体策略必须单一

本轮采用单一策略：

1. 原始响应体超过硬上限时直接失败，不再继续解码和提取
2. 只有在“响应体未超硬上限，但提取后的正文过长”时才允许截断
3. 发生正文截断时，成功结果中的 `truncated` 必须为真

## 11. 最终建议

采用 `A` 方案：

1. 保持 `fetch_url` 接口不变
2. 做请求层增强
3. 做编码与正文提取增强
4. 增加缓存
5. 增加错误分类

这是当前项目里性价比最高、风险最低、也最贴近你现在痛点的一条路线。
