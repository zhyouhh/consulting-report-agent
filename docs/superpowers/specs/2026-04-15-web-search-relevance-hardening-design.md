# Web Search 相关性加固设计稿

## 1. 背景

当前项目的 `web_search` 工具虽然已经接入工作流，但实际效果明显偏弱，已经影响到报告写作质量：

1. 对中文产业、公司、政策类问题，前几条结果经常混入明显无关的英文页面。
2. 结果返回后只被拼成一段纯文本，模型无法稳定地基于来源质量继续调用 `fetch_url`。
3. 搜索后端虽然是 SearXNG 风格的 JSON API，但当前调用几乎没有使用语言、分类、引擎等参数能力。
4. 搜索结果没有做去重、域名质量判断、来源优先级排序，导致垃圾站和聚合站容易排到前面。
5. 现有 SSH 凭据对应的服务器并不是 `search.z0y0h.work` 的实际部署主机，因此暂时无法直接改搜索实例配置。

本轮目标不是直接切换到 provider-native web search，而是在保留现有 `web_search + fetch_url` 工作流的前提下，把搜索召回与结果质量提升到“够用、稳定、少误导”的水平。

## 2. 已完成现网审计

### 2.1 应用侧现状

当前实现位于 [backend/chat.py](D:/CodexProject/Consult report/consulting-report-agent/backend/chat.py)：

1. `web_search` 作为自定义 function tool 暴露给模型。
2. 后端请求 `managed_search_api_url` 时仅传 `q` 和 `format=json`。
3. 返回结果后只抽 `title/content/url`，再拼成最多 5 条纯文本摘要。
4. 没有 query rewrite、参数约束、去重、重排或结构化结果。

### 2.2 黑盒 API 审计结论

对 `https://search.z0y0h.work/search` 的黑盒测试显示：

1. API 可正常返回 SearXNG 风格字段：`results`、`suggestions`、`infoboxes`、`unresponsive_engines`。
2. 中文查询 `"咨询行业 市场规模 中国 2025"` 在默认参数下，前几条结果可混入明显无关的英文页面。
3. `language=zh-CN` 并不能单独解决问题，说明问题不只在语言参数缺失，还在引擎池和结果排序。
4. `unresponsive_engines` 多次出现：
   - `baidu: CAPTCHA / Suspended: CAPTCHA`
   - `duckduckgo: CAPTCHA`
   - `brave: too many requests`
5. 对带 `site:` 约束的查询，相关性明显提升，说明 query rewrite 和域名约束能直接改善效果。

### 2.3 服务器侧审计结论

使用当前 SSH 凭据登录 `43.153.168.175:2233` 后确认：

1. 该机器上没有 `search.z0y0h.work` 的 nginx 站点配置。
2. 未发现 `searx` 或 `searxng` 容器/服务。
3. 因此当前无法直接修改搜索实例配置。

这意味着本轮需要拆成两部分：

1. 先做应用侧可立即落地的质量提升。
2. 服务器侧优化作为第二阶段，在拿到搜索实例真实主机后执行。

## 3. 目标

本次改造目标是：

1. 保持工具名 `web_search` 不变，不破坏现有模型工作流。
2. 提升中文主题搜索的前 3 到 5 条结果相关性。
3. 降低垃圾站、聚合站、英文无关页排到前列的概率。
4. 返回结构化结果，使模型更容易基于候选链接继续调用 `fetch_url`。
5. 与现有 “搜索后写正式文件前先 `fetch_url`” 的门禁保持一致。

## 4. 非目标

本次不做：

1. 不切换到 OpenAI / Gemini native web search。
2. 不重构整条 provider 通道。
3. 不引入多搜索源聚合系统。
4. 不要求所有查询都百分百命中高质量结果。
5. 在拿到真实搜索主机前，不直接改 `search.z0y0h.work` 服务配置。

## 5. 根因判断

当前搜索质量差，主要不是“XNG 完全不可用”，而是以下几个问题叠加：

### 5.1 查询构造过于原始

现在所有查询都裸传原始字符串，缺少：

1. 语言约束
2. 主题意图识别
3. 轻量 query rewrite
4. 域名约束或站点提示

### 5.2 结果后处理几乎为空

当前没有：

1. 同标题 / 同 URL 去重
2. 域名归一化
3. 来源质量分层
4. 中文场景的重排序

### 5.3 引擎健康状态未纳入策略

黑盒结果显示部分引擎频繁 CAPTCHA 或限流：

1. `baidu`
2. `duckduckgo`
3. `brave`

如果应用侧不识别这些状态，就会让低质量或异常回源结果继续污染前列排序。

### 5.4 结果输出格式太弱

当前把搜索结果压扁成纯文本，导致：

1. 模型不容易稳定选择最优链接
2. 后续 `fetch_url` 缺少结构化候选
3. 无法在内存侧保留可解释的来源信息

## 6. 方案概览

本轮采用 B 档方案，但拆为两个阶段。

### 阶段 A：立即可做的应用侧加固

在不改搜索服务配置的前提下，增强 `backend/chat.py::_web_search()`：

1. 查询意图识别
2. 轻量 query rewrite
3. SearXNG 参数增强
4. 搜索结果去重与重排
5. 结构化结果输出

### 阶段 B：拿到真实搜索主机后的服务侧调优

在确认真实部署主机后：

1. 盘点启用引擎
2. 清理明显噪声引擎
3. 区分中文 general/news/reference 搜索画像
4. 降低 CAPTCHA 高频引擎的默认权重或直接移除

## 7. 详细设计

### 7.1 查询意图识别

后端在接到 `query` 后，先做轻量规则判断，不暴露给模型：

1. `general`
   - 默认兜底
2. `news`
   - 含“最新 / 近期 / 发布 / 动态 / 新闻 / 通报 / 财报 / 收购”等词
3. `reference`
   - 含“是什么 / 简介 / 背景 / 历史 / 百科 / 定义 / 概念”等词
4. `company`
   - 含“公司 / 官网 / 年报 / 招股书 / 产品 / 业务 / 组织架构”等词

意图仅影响参数与重排，不改变工具接口。

### 7.2 轻量 Query Rewrite

不走复杂 LLM rewrite，只做规则型改写：

1. 中文产业研究类：
   - 可补 `市场规模`
   - 可补 `行业`
   - 可补 `中国`
2. 公司研究类：
   - 可补 `官网`
   - 可补 `年报`
   - 可补 `招股书`
3. 概念/背景类：
   - 可补 `百科`
   - 可补 `简介`
4. 政策类：
   - 可补 `原文`
   - 可补 `政府`
   - 可补 `政策`

规则：

1. 仅追加一个辅助查询，不生成复杂查询树。
2. 主查询和辅助查询都可打搜索，但合并时统一去重和重排。
3. 若原查询已包含明显站点约束或强意图词，不再重复改写。

### 7.3 SearXNG 参数增强

应用侧改为按意图构造参数：

1. 默认补 `language=zh-CN`
2. `news` 意图尝试 `categories=news`
3. `reference` 意图可尝试偏百科/知识源引擎池
4. 支持选择有限引擎集合，而不是所有默认引擎全开

注意：

1. 如果某组引擎在该实例上经常限流，应用侧应允许降级到备选引擎组合。
2. 不依赖单一引擎成功。

### 7.4 结果去重

去重规则：

1. 相同规范化 URL 去重
2. 相同标题 + 同主域去重
3. 同一站点的分页/镜像页保留最优一条

URL 规范化至少处理：

1. 去 fragment
2. 去常见追踪参数
3. 统一大小写 host

### 7.5 域名质量重排

为每条结果打轻量分数，按以下因素加减权：

加权来源：

1. 官网
2. 政府站 `gov.cn`
3. 统计/协会/高校/研究机构
4. 百科类权威页面
5. 主流媒体或成熟行业站

降权来源：

1. 内容农场
2. 纯采集站
3. 低质视频/动漫/影视聚合站
4. 与查询语言显著不匹配的页面

如果标题或摘要与查询关键词重合度过低，也降权。

### 7.6 结构化结果输出

对模型返回时，不再只提供纯文本，而是同时保留结构化字段：

1. `title`
2. `snippet`
3. `url`
4. `domain`
5. `engine`
6. `score`
7. `reason_tags`

同时继续保留兼容的文本摘要，避免现有模型提示完全失配。

### 7.7 与 fetch_url 门禁对齐

搜索工具的职责继续保持为“找候选”，不直接充当正式依据。

规则继续保留：

1. 如果本轮做过 `web_search`
2. 且准备把外部信息写入正式文件
3. 则必须至少对候选链接调用 `fetch_url`

这一点不改，只让候选更靠谱。

## 8. 服务侧调优预案

在拿到真实搜索主机后，服务器侧建议按以下顺序执行：

1. 备份当前 SearXNG 配置
2. 导出当前启用引擎和超时配置
3. 检查中文场景默认引擎池
4. 移除或降权 CAPTCHA 高频引擎
5. 为中文场景定义更稳的引擎组合
6. 用固定查询集做前后对比

目标不是“引擎越多越好”，而是：

1. 少而稳
2. 对中文主题相关性更高
3. 与 `fetch_url` 成功率更匹配

## 9. 验证方案

验证集至少包含三类查询：

1. 公司/行业研究
2. 政策/统计数据
3. 百科/背景事实

每类用 5 到 10 条固定查询，对比：

1. 前 3 条相关性
2. 是否更容易进入 `fetch_url`
3. 是否明显减少无关英文页
4. 是否明显减少垃圾站

## 10. 风险与回滚

风险：

1. 查询改写过强，可能把原始意图带偏
2. 域名重排过猛，可能漏掉有价值长尾来源
3. 服务器侧引擎池调整若无灰度，可能误伤已有可用结果

回滚策略：

1. 应用侧保留旧版搜索拼接逻辑开关
2. 服务器侧变更前先备份配置
3. 先应用侧上线，再等待真实搜索主机确认后做服务侧调整

## 11. 结论

本轮最合理的路径不是直接换成 native web search，而是：

1. 先把现有 `web_search` 的查询构造、结果重排和结构化输出做对
2. 再在拿到真实 SearXNG 主机后，补上服务侧引擎池治理

这条路径最符合当前项目状态，也最符合奥卡姆剃刀：先修真正的薄弱点，不引入新系统。
